Reference architecture · Claude Code × WhatsApp

Autonomous WhatsApp on Claude Code — the brain / transport pattern

The public repos all solve the same thing: you talking to your own Claude Code from your phone. None of them solve the harder problem — an autonomous agent answering strangers on a business line, safely. Here's the topology that does.

TL;DR

Stop fusing the model and the wire. Split them: Claude Code (Opus) is the brain — it drafts, reasons, remembers, and never touches the WhatsApp socket. A separate linked-device client is a dumb transport — it carries bytes and never thinks. Between them sits a governance loop (context → draft → shadow → approve → token-gate → send) that makes stranger-facing autonomy safe. The connection is commoditized; the orchestration above it is the IP.

01The landscape: three ways people already wired CC to WhatsApp

WhatsApp as a tool

An MCP server exposes WA read/send as tool calls. Claude queries your chats — "search my messages", "send this". WA is a data source the model reaches into.

02What's unsolved publicly

Flip the arrow. The hard problem isn't reaching your own agent — it's letting your agent reach everyone else.

Solved & commoditized
  • Linked-device pairing (Baileys / WhatsApp Web, no Business API)
  • Operator ↔ agent bidirectional chat
  • Voice transcription, media, group routing
  • Remote tool-approval from the phone
Open — nobody has packaged this
  • Autonomous inbound from third-party strangers — clients, suppliers, prospects messaging a business line
  • Business-correct drafts grounded in per-contact memory + enrichment
  • A governance loop so the model never fires raw output at a real customer
  • Separation of the thinking from the wire

03The pattern: a two-tier split

One process reasons. A different process carries bytes. They share nothing but a queue.

Brain — the only thing that thinks Transport — the only thing on the wire Queue — the seam between them

04The governance loop

Every inbound message walks this path. The model's words don't reach a human until they clear shadow, approval, and an outbound token gate.

On the wire (transport) Thinking (brain) Hard gate Approved & sent

05Why the split matters

Fusing brain and transport is fine for talking to yourself. It's a liability the moment the other party is a stranger.

01Separation of concerns

The brain can be restarted, upgraded, or swapped models without dropping the WhatsApp session. The transport can reconnect after a pairing drop without losing a single draft. Neither failure cascades.

02A blast radius you can reason about

The transport has no judgment and no creativity — so it can't hallucinate a promise to a customer. The brain has no socket — so a prompt-injection in an inbound message can't make it send. Capability is split from authority.

03Autonomy you can actually grant

Because the guards are structural, not vibes, you can dial autonomy up per contact or intent — full-auto for low-stakes, shadow-only for high-stakes — without rewriting the agent. The loop is the same; only the gate threshold moves.

04The connection is replaceable

The transport is a commodity — any linked-device client works. Your moat is the brain layer: contact memory, enrichment, the governance loop. Build on someone else's wire; keep the orchestration as yours.

That's Cluster B, and it's a solved problem — don't rebuild it. Clone the reference and you're paired in minutes.
